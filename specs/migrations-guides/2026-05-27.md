# Migration Guide: Legacy CDM SDK to `main`

Date: 2026-05-27

Audience: an implementation agent migrating CDM consumers and system contracts from the old experimental `charles/cdm-integration` branch of `cargo-pvm-contract` to the current SDK/storage work on `main`.

Make sure to deeply look at the main branch especially to see if there are new features present which this guide might say are not present at the time of writing.

## Goal

Move CDM packages and contracts onto the current `cargo-pvm-contract` SDK shape without rewriting the registry contract yet.

The legacy branch carried CDM-specific behavior directly in the SDK and macro system. Over the last few months the same broad feature set has been rebuilt in a cleaner split:

- `cargo-pvm-contract` owns the contract SDK, ABI import, contract macro, storage, and build command.
- `contract-dependency-manager` owns CDM-specific package identity, package resolution, `cdm::import!`, and `pvm-cdm`.

For the current migration, do not assume a contract still compiles just because it built on `charles/cdm-integration`. The new SDK intentionally changed the contract shape, storage API, and CDM identity source.

## Start By Reading

Clone both SDK branches into normal reference directories and compare them locally:

```sh
mkdir -p reference-repos

git clone https://github.com/paritytech/cargo-pvm-contract.git reference-repos/cpc-main
git -C reference-repos/cpc-main checkout main

git clone https://github.com/paritytech/cargo-pvm-contract.git reference-repos/cpc-legacy-cdm
git -C reference-repos/cpc-legacy-cdm checkout charles/cdm-integration
```

Review the relevant upstream discussions before making broad assumptions:

- `paritytech/cargo-pvm-contract#59`: CDM integration proposal. This is where CDM identity moved out of the SDK and into `Cargo.toml`.
- `paritytech/cargo-pvm-contract#73`: storage improvements for composition and dynamic values. This is the current storage model to target.
- `paritytech/cargo-pvm-contract#80`: original issue for dynamic/composite storage blockers.
- `paritytech/cargo-pvm-contract#84`: streamed JSON build output and accurate build-plan totals for CDM progress UI.

## Why Old Contracts May Disappear From `cdm deploy`

If `cdm deploy -n paseo` connects to the chains and prints the registry address but no contract rows, the most likely cause is that no workspace package is detected as a current CDM contract.

Current CDM detection uses `cargo metadata` and only includes workspace packages that:

- depend on `pvm-contract-sdk` or legacy `pvm_contract`,
- have a binary target,
- declare a CDM package name in Cargo metadata.

The current CDM identity source is:

```toml
[package.metadata.cdm]
package = "@polkadot/contexts"
```

The legacy source was:

```rust
#[pvm::contract(cdm = "@polkadot/contexts")]
```

Contracts that still only use the legacy macro argument will be skipped by current CDM tooling.

## Dependency Changes

Replace the old SDK dependency:

```toml
pvm_contract = { git = "https://github.com/paritytech/cargo-pvm-contract", branch = "charles/cdm-integration" }
```

with the current SDK and CDM crates:

```toml
[workspace.dependencies]
cdm = { git = "https://github.com/paritytech/contract-dependency-manager", branch = "main" }
pvm-contract-sdk = { git = "https://github.com/paritytech/cargo-pvm-contract", branch = "main", features = ["alloc"] }
pvm-cdm = { git = "https://github.com/paritytech/contract-dependency-manager", branch = "main" }
polkavm-derive = "0.31"
picoalloc = "5.2"
```

Each contract package should have both a library and a binary target:

```toml
[lib]
path = "lib.rs"

[[bin]]
name = "contexts"
path = "lib.rs"

[features]
abi-gen = ["pvm-contract-sdk/abi-gen"]

[package.metadata.cdm]
package = "@polkadot/contexts"
```

For local contract-to-contract dependencies, keep normal Cargo path dependencies for build order even when source code uses `cdm::import!`:

```toml
[dependencies]
cdm = { workspace = true }
pvm-contract-sdk = { workspace = true }
pvm-cdm = { workspace = true }
contexts = { path = "../contexts" }
```

The source import should still be:

```rust
cdm::import!("@polkadot/contexts");
```

`cdm::import!` resolves a workspace-local ABI first by matching `[package.metadata.cdm].package`, then falls back to the installed ABI from `cdm.json`.

## Contract Macro Shape

Legacy contracts are usually written as free functions plus a separate `#[pvm::storage]` struct:

```rust
#[pvm::storage]
struct Storage {
    context_owner: Mapping<ContextId, Address>,
}

#[pvm::contract(cdm = "@polkadot/contexts")]
mod contexts {
    #[pvm::method]
    pub fn get_owner(context_id: ContextId) -> Address {
        Storage::context_owner().get(&context_id).unwrap_or_default()
    }
}
```

Current contracts declare storage fields on the contract struct and implement methods on that struct:

```rust
#![cfg_attr(not(feature = "abi-gen"), no_main, no_std)]

#[pvm_contract_sdk::contract(allocator = "pico", allocator_size = 1024)]
mod contexts {
    use pvm_contract_sdk::{Address, Mapping};
    use common::ContextId;

    pub struct Contexts {
        #[slot(0)]
        context_owner: Mapping<ContextId, Address>,
    }

    impl Contexts {
        #[pvm_contract_sdk::constructor]
        pub fn new(&mut self) {}

        #[pvm_contract_sdk::method]
        pub fn get_owner(&self, context_id: ContextId) -> Address {
            self.context_owner.get(&context_id)
        }
    }
}
```

Use `&self` for view methods and `&mut self` for methods that write storage. No receiver means pure.

## Host Calls And Reverts

Legacy helpers like `pvm::caller()` and `pvm::api::now()` need to move to the current host API:

```rust
use pvm_contract_sdk::{Address, ContractContext, HostApi};

fn caller<C: ContractContext>(contract: &C) -> Address {
    let mut bytes = [0u8; 20];
    contract.host().caller(&mut bytes);
    Address(bytes)
}
```

For timestamps:

```rust
let mut buf = [0u8; 32];
self.host().now(&mut buf);
let timestamp = u64::from_le_bytes(buf[0..8].try_into().unwrap_or([0u8; 8]));
```

For `common::revert`, either update the helper to accept a host/context or keep small local helpers inside each contract module. The important change is that reverts now go through `HostApi::return_value` with `ReturnFlags::REVERT`; do not depend on the old global `pvm::revert` shape.

## Storage Semantics

The current `main` storage layer is not the early single-slot-only layer. It supports:

- `Lazy<T>` and `Mapping<K, V>` for primitive values.
- Multi-slot static structs.
- Structs with `String` and Solidity `bytes` fields.
- Dynamic keys such as `String` and `Vec<u8>` mapping keys.
- Nested storage components via `#[pvm_contract_sdk::storage]`.

For storage values, derive `SolType`:

```rust
#[derive(Default, Clone, pvm_contract_sdk::SolType)]
pub struct RunningAverage {
    pub sum: u64,
    pub total: u64,
}

#[derive(Default, Clone, pvm_contract_sdk::SolType)]
pub struct Review {
    pub reviewer: pvm_contract_sdk::Address,
    pub rating: u8,
    pub comment_uri: alloc::string::String,
}
```

Then these are valid on current `main`:

```rust
metrics: Mapping<(ContextId, EntityId), RunningAverage>,
reviews: Mapping<(ContextId, EntityId, u64), Review>,
```

Important API differences:

- `Mapping<K, V>::get(&key)` returns `V`, with zero/default storage semantics.
- `Mapping<K, V>::try_get(&key)` returns `Option<V>` when absence must be distinguished.
- `insert` and `remove` require `&mut self`.
- There is no legacy `contains` method. Use `try_get(&key).is_some()` or store a `bool` value and use `get(&key)`.
- `Vec<u8>` is not a storage value. Use `pvm_contract_sdk::Bytes` for Solidity `bytes`.
- Arbitrary `Vec<T>` is not a storage value or storage struct field yet.

## CDM References

Legacy imported contracts generated `contexts::Reference` and `contexts::cdm_reference()`.

The current flow is:

```rust
cdm::import!("@polkadot/contexts");

let contexts = contexts::Contexts::cdm_lookup();
let allowed = contexts
    .is_authorized(context_id, caller)
    .call(self)
    .expect("contexts call failed");
```

`cdm_lookup()` resolves through the runtime registry using `CONTRACTS_REGISTRY_ADDR`, which CDM sets during build/deploy. `cdm_from_env()` bakes addresses from `CDM_REGISTRY` for tests or frozen builds.

Do not assume the new imported reference type is safe to cache directly in storage. If caching is needed, store the raw `Address` or re-check whether the generated reference type implements the current storage traits. The simplest first migration is to call `cdm_lookup()` when needed.

## Current Contract-Specific Notes

### `common`

Update shared ABI/storage helper types first. `RunningAverage` should derive `pvm_contract_sdk::SolType` so it can be used inside `Mapping<K, RunningAverage>`.

Remove `parity-scale-codec` from storage-facing types unless another caller still needs it. The current storage layer is Solidity-layout based, not SCALE-layout based.

### `contexts`

Likely straightforward.

Replace:

```rust
context_operators: Mapping<(ContextId, Address), ()>,
```

with:

```rust
context_operators: Mapping<(ContextId, Address), bool>,
```

Store `true` when an operator is present. `get(&key)` returns `false` for absent storage, which fits authorization checks.

### `reputation`

This is a good first migration target.

`Review { reviewer, rating, comment_uri: String }` and `RunningAverage` should now be usable as storage values when they derive `SolType`. The old dynamic/composite storage blocker should be gone on `main`.

Replace the stored `contexts::Reference` with `cdm::import!("@polkadot/contexts")` plus `Contexts::cdm_lookup()` at call time unless there is a concrete reason to cache an address.

### `profiles`

`ProfileData { owner, metadata_uri: String }` should be usable as storage after deriving `SolType`.

The blocker is ABI return shape: current `main` does not appear to have the legacy branch's intrinsic `Option<T>` ABI support. Existing methods like these need redesign or upstream support:

```rust
get_profile_info(...) -> Option<Profile>
get_profile_at(...) -> Option<EntityId>
```

Use an explicit return struct such as `{ exists: bool, value: T }`, split into `exists` plus `get`, or wait for `Option<T>` ABI support to land.

### `threads`

There are two blockers:

- `PostData` stores `parents: Vec<EntityId>`. Arbitrary `Vec<T>` is not supported as a storage value/field on current `main`.
- Several ABI methods return `Option<T>`, which appears unsupported today.

Keep `Vec<T>` for ABI params/returns where needed; the issue is storing it. For storage, flatten parents into `parent_count` / `parent_at` mappings or a similar indexed shape. This contract already has parent indexes, so avoid duplicating a large vector inside the `PostData` storage record.

### `disputes`

Likely feasible after the shared and contexts migrations.

`Dispute` and `Instruction` contain `String` fields but no arbitrary `Vec<T>` fields, so they should work as storage values with `SolType` derives. Verify whether any public methods return `Option<T>` before migrating.

### Registry Contract

Do not migrate the registry contract as part of this pass unless explicitly requested. It is still written against the old branch, and the migration objective here is CDM consumers/packages plus selected system contracts.

## Remaining Upstream Gaps To Track

These are real gaps relative to the old experimental branch:

- `Option<T>` ABI support is not obviously present on current `main`.
- Arbitrary `Vec<T>` / Solidity array storage is not supported. `Vec<T>` remains valid for ABI dynamic arrays.
- `OrderedIndex` was implemented on the old experimental branch. Do not depend on it on `main` unless it is ported or rebuilt on top of the new storage traits.

These are no longer expected blockers on current `main`:

- `String` storage values.
- Solidity `bytes` storage values when represented as `pvm_contract_sdk::Bytes`.
- Multi-slot static structs.
- Structs with dynamic `String` / `Bytes` fields.

## Build Tooling

Install cargo-pvm-contract from the main branch for local CLI use:

```sh
HOST_TARGET=$(rustc -vV | awk '/^host:/ {print $2}')
cargo install --force --locked \
  --target "$HOST_TARGET" \
  --git https://github.com/paritytech/cargo-pvm-contract.git \
  --branch main \
  cargo-pvm-contract
```

If CDM build progress needs a determinate bar, make sure the installed `cargo-pvm-contract` includes PR #84 or a `main` merge that contains it. That PR streams Cargo JSON and emits a namespaced `cargo-pvm-contract-build-plan` event with an accurate `compiler-artifact` total.

Direct build command shape:

```sh
cargo pvm-contract build \
  --manifest-path Cargo.toml \
  -p reputation \
  --message-format json,json-diagnostic-rendered-ansi
```

Use `cdm build` and `cdm deploy` for normal workflow once package metadata and local path dependencies are in place.

## Verification Checklist

For each migrated workspace:

- `cargo metadata --format-version 1 --no-deps` shows `[package.metadata.cdm].package` under each contract package.
- `cdm build` lists the expected contracts in dependency order.
- Contract dependencies are represented as Cargo path dependencies when they are local workspace contracts.
- `cdm::import!("@scope/name")` resolves local ABIs after a `cdm build` without requiring publish/install.
- `cdm deploy -n paseo` prints rows for each deployable contract.
- Cross-contract calls use `cdm_lookup()` and the target registry address printed by CDM is the intended one.
- No migrated storage struct contains arbitrary `Vec<T>` fields.
- No public ABI method returns `Option<T>` unless upstream support has landed.

## Do Not Paper Over These

- Do not keep `#[pvm::contract(cdm = "...")]`; that is the legacy CDM identity path.
- Do not store `Vec<T>` in a `Mapping` value or `Lazy` value on current `main`.
- Do not manually flatten `Review` or `RunningAverage` only because of old storage limitations; current `main` should store those as composite values.
- Do not remove local Cargo path dependencies just because `cdm::import!` names the CDM package. CDM still uses Cargo metadata for dependency order.
- Do not commit migration work without the user's GPG/YubiKey signing flow.
